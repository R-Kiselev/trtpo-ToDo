# Отчёт по Лабораторной работе №5
**«Технический долг. Устранение долга»**

---

## 1. Введение

Технический долг — это совокупность архитектурных и кодовых недостатков, накапливающихся в процессе разработки под давлением сроков и неполноты требований. Чем дольше долг остаётся без внимания, тем выше стоимость его устранения и тем сложнее дальнейшее развитие продукта.

В рамках данной лабораторной работы была проведена инспекция проекта (микросервисы `CoreService` и `AuthService` на Python/FastAPI), выявлены признаки технического долга, разработан план мероприятий по его устранению, проведена оценка плана и сделаны выводы о необходимости устранения долга.

---

## 2. Инспекция проекта на наличие технического долга

### 2.1. Непонятный / нечитабельный код

**Наличие:** Частичное.

В `CoreService` и `AuthService` в ряде мест встречались функции с размытой ответственностью, совмещающие логику обработки запроса, валидацию и работу с базой данных в одном месте. Часть переменных имела неинформативные названия (`data`, `res`, `obj`), что снижало читаемость при переключении между задачами.

**Пример:** В `AuthService` логика генерации JWT-токена частично была смешана с логикой валидации входящего запроса в одном роутере, вместо того чтобы быть вынесенной в отдельный сервисный слой.

### 2.2. Дублирующийся код

**Наличие:** Есть.

Обработка HTTP-исключений повторялась в нескольких роутерах `CoreService` без вынесения в общий exception handler. Аналогичная ситуация наблюдалась с валидацией входящих Pydantic-схем — часть проверок дублировалась на уровне роутера и сервисного слоя.

**Вывод:** Нарушение принципа DRY (Don't Repeat Yourself), что повышает риск рассинхронизации логики при будущих изменениях.

### 2.3. Отсутствие автоматизации (тестов, сборки, развёртывания)

**Наличие:** Частично устранено.

Базовые GitHub Actions были настроены для автоматической проверки кода. Однако покрытие тестами отдельных компонентов `AuthService` оставалось ниже 60%, а для ряда граничных сценариев (истёкший токен, неверный формат токена, попытка повторной регистрации) тесты отсутствовали полностью.

### 2.4. Запутанная архитектура и ненужные сложные зависимости

**Наличие:** Незначительное.

В `CoreService` наблюдалась прямая зависимость между роутерами и слоем работы с базой данных без промежуточного сервисного слоя. Это затрудняло написание изолированных unit-тестов, так как для тестирования одной функции требовалось поднимать подключение к базе данных.

### 2.5. Медленные / неэффективные средства

**Наличие:** Незначительное.

В ряде эндпоинтов `CoreService` использовались синхронные запросы к базе данных вместо асинхронных, что при высокой нагрузке снижает пропускную способность FastAPI-приложения, изначально рассчитанного на async/await модель.

### 2.6. Незакоммиченный код / долгоживущие ветки

**Наличие:** Было зафиксировано.

В процессе работы над спринтом несколько раз незавершённые изменения оставались в рабочей копии без коммита на протяжении нескольких часов. Также была выявлена ветка `feature/auth-refactor` в GitHub, которая не мержилась в `main` около двух недель, что создавало риск конфликтов и расхождения с актуальным состоянием кодовой базы.

### 2.7. Отсутствие / несоответствие технической документации

**Наличие:** Частичное.

FastAPI автоматически генерирует Swagger-документацию, однако эндпоинты `CoreService` и `AuthService` не имели описаний (`summary`, `description`), примеров запросов и ответов, а также описаний кодов ошибок. Документация была формально доступна, но не несла практической пользы для нового разработчика.

### 2.8. Отсутствие тестовой среды

**Наличие:** Частично.

Изолированная тестовая среда с тестовой базой данных была настроена, однако выделенного окружения с предзаполненными тестовыми данными для ручной проверки не существовало. При каждом новом запуске локального окружения базу данных приходилось наполнять вручную.

### 2.9. Длинные циклы интеграции / отсутствие непрерывной интеграции

**Наличие:** Незначительное.

GitHub Actions настроен и работает. Тем не менее из-за последовательного (а не параллельного) запуска этапов линтинга, тестирования и сборки полный цикл занимал больше времени, чем необходимо, что замедляло получение обратной связи после каждого коммита.

## 3. План мероприятий по устранению технического долга

| № | Мероприятие | Сервис | Приоритет | Оценка (ч) |
|---|-------------|--------|-----------|------------|
| 1 | Вынести общий exception handler, устранить дублирование обработки ошибок в роутерах | `CoreService` | Высокий | 2 |
| 2 | Добавить `summary`, `description` и примеры ответов к эндпоинтам в Swagger | `CoreService`, `AuthService` | Высокий | 3 |
| 3 | Написать недостающие тесты для граничных случаев JWT (истёкший, невалидный токен, повторная регистрация) | `AuthService` | Высокий | 4 |
| 4 | Вынести логику генерации и валидации токена из роутера в отдельный сервисный слой | `AuthService` | Средний | 3 |
| 5 | Перевести синхронные запросы к БД на async в `CoreService` | `CoreService` | Средний | 3 |
| 6 | Ввести правило обязательного мержа веток не позднее 7 дней | Процесс | Средний | 0.5 |
| 7 | Настроить параллельный запуск этапов в GitHub Actions | Инфраструктура | Средний | 1 |
| 8 | Подготовить скрипт предзаполнения тестовой базы данных (data seeding) | Инфраструктура | Низкий | 2 |

## 4. Оценка плана мероприятий

Суммарная оценка устранения выявленного долга составила **~20.5 часа** при среднем темпе команды 2 разработчика. При распределении по приоритетам:

- **Высокий приоритет:** ~9 часов — устраняет наиболее критичные риски: дыры в тестовом покрытии `AuthService`, дублирование обработки ошибок, отсутствие полезной документации.
- **Средний приоритет:** ~7.5 часов — снижает операционные издержки, улучшает архитектуру и скорость CI.
- **Низкий приоритет:** ~4 часа — улучшает качество жизни команды, но не критичен на текущем этапе.

Если долг не устранять, при добавлении каждой новой фичи вероятность регрессий в `AuthService` будет расти, а отсутствие нормальной документации будет замедлять онбординг и увеличивать время на ревью.

## 5. Сравнение объёма долга и недоимплементированных фич

На момент окончания спринта в бэклоге оставались следующие незакрытые пользовательские истории:

- **US-6:** Ролевая модель (разграничение прав доступа) — оценка ~10 часов.
- **US-7:** Уведомления пользователей — оценка ~8 часов.
- **US-8:** Расширенная фильтрация и поиск в `CoreService` — оценка ~6 часов.

**Суммарный объём незакрытых фич:** ~24 часа.  
**Суммарный объём технического долга:** ~20.5 часа.  
**Соотношение:** технический долг составляет ~85% от объёма оставшихся фич.

Это пограничная ситуация: долг достаточно велик, чтобы его нельзя было игнорировать, но полная остановка разработки ради рефакторинга нецелесообразна. Рекомендуется применить подход **«Definition of Done»**: включить критерии чистоты кода (отсутствие дублирования, наличие документации, покрытие тестами >80%) в условие закрытия каждой следующей User Story.

## 6. Выводы и обоснование необходимости устранения долга

### Необходимость устранения

Устранение технического долга в данном проекте **частично необходимо** — не в полном объёме и не немедленно, но ключевые позиции требуют действий уже в следующем спринте.

**Обоснование:**

1. **Критичные позиции** (пункты 1–3 плана) напрямую влияют на надёжность `AuthService` и качество API. Отсутствие тестов на граничные случаи работы с токенами — реальный риск безопасности, который нельзя откладывать.
2. **Средние позиции** (пункты 4–7) снижают операционные издержки и улучшают архитектуру. Их устранение окупается за 1–2 спринта.
3. **Низкий приоритет** (пункты 8–9) можно планомерно закрывать в рамках обычных задач, не выделяя отдельного времени.

### Корректировки плана разработки

- В следующий спринт включить технические задачи: exception handler, Swagger-документация, тесты для `AuthService` — суммарно ~9 часов.
- Ввести правило: любая новая фича сдаётся только при покрытии тестами >80% и заполненной Swagger-документации для новых эндпоинтов.
- Параллельный GitHub Actions и правило по веткам — задача на 1.5 часа, выполняется в рамках дежурства по инфраструктуре.

### Общий вывод

Проект находится в состоянии допустимого технического долга, не превышающего критический уровень. Полная остановка разработки для его устранения нецелесообразна. Оптимальная стратегия — **параллельное сокращение долга** через Definition of Done, точечные технические задачи в спринте и постепенный рефакторинг в рамках работы над новыми фичами. При соблюдении этого подхода к концу следующего спринта ключевые позиции долга будут закрыты, а проект сохранит темп разработки.
