# Отчёт по Лабораторной работе №6
**«Улучшение UX»**

---

## 1. Введение и цели работы

Целью данной лабораторной работы является качественный аудит и последующее улучшение пользовательского интерфейса (UI) и пользовательского опыта (UX) проекта. Работа опирается на международный стандарт ISO/IEC 25010:2011, спецификации доступности WCAG 2.0 и принципы многоуровневого проектирования опыта использования.

Проект представляет собой веб-приложение на базе двух микросервисов (`CoreService` и `AuthService`), разработанных на Python/FastAPI. Переход от функционального MVP к полноценному продукту требует фокуса на эмпатии к пользователю, минимизации когнитивной нагрузки и предотвращении критических ошибок при работе с API и интерфейсом.

## 2. Оценка разрабатываемого ПО по атрибутам качества «Usability» (ISO/IEC 25010:2011)

До проведения работ по улучшению приложение было проанализировано по шести ключевым атрибутам качества группы Usability (SQuaRE):

1. **Распознаваемость соответствия (Appropriateness recognizability) — 4/5**
   - *Оценка:* Пользователь с первых секунд понимает назначение продукта: форма авторизации (`AuthService`) содержит стандартные поля логина и пароля с понятными подписями. Основной интерфейс (`CoreService`) предоставляет доступ к задачам и данным через интуитивно понятную структуру. Незначительный минус — отсутствие описания назначения продукта на стартовой странице для новых пользователей.

2. **Обучаемость (Learnability) — 2/5 (ДО) ➔ 5/5 (ПОСЛЕ)**
   - *Проблема:* При первом входе в систему пользователь видел пустой список задач без каких-либо подсказок. Интерфейс не объяснял, что нужно сделать, чтобы начать работу. Новый пользователь не понимал, пуста ли база данных или произошла ошибка загрузки.
   - *Решение:* Внедрение паттерна «Empty State» с иконкой, поясняющим текстом и явной кнопкой призыва к действию (Call-to-Action): «Задач пока нет. Нажмите "Создать", чтобы добавить первую».

3. **Используемость / Операбельность (Operability) — 3/5 (ДО) ➔ 5/5 (ПОСЛЕ)**
   - *Проблема:* Форма авторизации и основные элементы управления не поддерживали навигацию с клавиатуры. Отсутствовала возможность отправить форму логина нажатием `Enter`, переключаться между полями клавишей `Tab` и управлять интерактивными элементами без мыши.
   - *Решение:* Добавлена полноценная поддержка клавиатурной навигации: обработчик события `keydown` для клавиши `Enter` в форме логина, корректный порядок `tabindex` для всех интерактивных элементов.

4. **Защита от ошибок пользователя (User error protection) — 2/5 (ДО) ➔ 5/5 (ПОСЛЕ)**
   - *Проблема:* Форма авторизации (`AuthService`) не давала мгновенной обратной связи при вводе некорректных данных. Пользователь нажимал «Войти», получал ответ `401 Unauthorized` от API, но интерфейс либо ничего не отображал, либо показывал технический текст ошибки. Форма не валидировала пустые поля до отправки запроса.
   - *Решение:* Реализована клиентская валидация формы до отправки запроса на сервер: проверка на пустые поля, минимальную длину пароля (не менее 6 символов) и корректный формат email. Ошибки отображаются inline непосредственно под соответствующим полем с понятным текстом на русском языке.

5. **Эстетика GUI (User interface aesthetics) — 3/5 (ДО) ➔ 5/5 (ПОСЛЕ)**
   - *Проблема:* Интерфейс использовал стандартные HTML-элементы без единого визуального стиля. Сообщения об ошибках отображались в виде браузерных `alert()`, которые нарушали общую эстетику и выглядели как системные всплывающие окна.
   - *Решение:* Стандартные `alert()` заменены на кастомные уведомления (toast-notifications) с анимацией появления и автоматическим скрытием. Внедрена единая цветовая схема: зелёный для успешных операций, красный для ошибок, жёлтый для предупреждений.

6. **Доступность (Accessibility / WCAG 2.0) — 1/5 (ДО) ➔ 5/5 (ПОСЛЕ)**
   - *Проблема:* Кнопки управления (редактировать, удалить, закрыть форму) состояли исключительно из иконок без текстовых подписей. Программы экранного доступа (Screen Readers) озвучивали их как «Пустая кнопка». Поля формы не имели корректно привязанных `label`.
   - *Решение:* Ко всем иконочным кнопкам добавлены атрибуты `aria-label` с описанием действия. Поля форм обёрнуты в корректные `<label>`. В CSS добавлен псевдокласс `:focus-visible` для визуального выделения активного элемента при навигации с клавиатуры.

## 3. Декомпозиция по уровням «Опыта использования» (Jesse James Garrett's 5 Elements of UX)

### Уровень 1: Стратегии (Strategy)

- **Потребности пользователей:** Быстро авторизоваться в системе, получить доступ к своим данным, добавить или обновить запись, получить понятный отклик от интерфейса при любом действии.
- **Бизнес-цели:** Минимизация времени на освоение продукта (Time-to-Value), снижение количества обращений в поддержку за счёт интуитивного интерфейса и информативных сообщений об ошибках.

### Уровень 2: Набора возможностей (Scope)

- **Функционал:** Регистрация и авторизация пользователей (`AuthService`), выдача и валидация JWT-токенов, CRUD-операции над сущностями (`CoreService`), фильтрация и поиск по данным, информативная обработка ошибок API, адаптивный интерфейс.

### Уровень 3: Структуры (Structure)

- **Информационная архитектура:** Двухэкранная модель. Экран авторизации (`/login`, `/register`) линейно переходит в основной рабочий экран (`/dashboard`), где доступны все операции с данными. Навигация минимальна, что снижает показатель Interaction Cost. Ошибки API перехватываются на уровне HTTP-клиента и транслируются в понятные пользователю сообщения.

### Уровень 4: Компоновки (Skeleton)

- **Расположение элементов:**
  - *Header:* Идентификация текущего пользователя и кнопка выхода из системы.
  - *Main Area:* Основной список данных с поддержкой фильтрации и сортировки.
  - *Action Panel:* Форма создания и редактирования записей, доступная без перехода на отдельную страницу.
  - *Notification Area:* Зона отображения toast-уведомлений об успехе или ошибке операции.

### Уровень 5: Поверхности (Surface)

- **Визуальный дизайн:** Нейтральная цветовая палитра с акцентными цветами для статусов (зелёный — успех, красный — ошибка, жёлтый — предупреждение). Чёткая типографика для читаемости данных. Единообразные стили для кнопок первичного и вторичного действия.

## 4. Обоснование и реализация UX-улучшений («ДО» vs «ПОСЛЕ»)

### Улучшение №1: Клиентская валидация формы авторизации

- **Необходимость:** Отправка каждого некорректного запроса на сервер создаёт лишнюю нагрузку и увеличивает время ответа пользователю. Отсутствие inline-сообщений об ошибках вынуждает пользователя угадывать, что именно он ввёл неверно.
- **Реализация:** Добавлена функция `validateForm()`, которая срабатывает до отправки запроса к `AuthService`. Проверяются: заполненность полей, минимальная длина пароля, корректность формата email.
- **ДО:** Форма отправляла пустые или некорректные данные на сервер, получала `422 Unprocessable Entity` или `401 Unauthorized`, после чего ничего не происходило или показывался технический текст.
- **ПОСЛЕ:** Пустое поле подсвечивается красной рамкой, под полем появляется текст: «Поле обязательно для заполнения» или «Пароль должен содержать не менее 6 символов». Запрос к API не отправляется до исправления ошибок.

### Улучшение №2: Замена `alert()` на toast-уведомления

- **Необходимость:** Системные `alert()` блокируют интерфейс, выглядят по-разному в разных браузерах и нарушают визуальный стиль. Они не дают пользователю продолжить работу до явного подтверждения.
- **Реализация:** Создан компонент `showToast(message, type)`, который добавляет в DOM уведомление с анимацией появления и автоматически скрывает его через 3 секунды.
- **ДО:** При успешном входе или ошибке появлялось блокирующее браузерное окно с техническим текстом.
- **ПОСЛЕ:** В правом верхнем углу экрана ненавязчиво появляется цветное уведомление: «Вход выполнен успешно» (зелёное) или «Неверный логин или пароль» (красное). Интерфейс остаётся доступным.

### Улучшение №3: Empty State для пустого списка данных

- **Необходимость:** Пустой экран без объяснений вызывает фрустрацию у нового пользователя. Интерфейс должен явно подсказывать следующий шаг.
- **Реализация:** Добавлена проверка: если список данных, возвращённый `CoreService`, пуст, основной контейнер скрывается и отображается блок `empty-state` с иконкой и кнопкой призыва к действию.
- **ДО:** При отсутствии данных основная область экрана оставалась пустой без каких-либо объяснений.
- **ПОСЛЕ:** Отображается блок с пунктирной рамкой и текстом: «Здесь пока пусто. Нажмите "Создать", чтобы добавить первую запись».

## 6. Выводы

В ходе выполнения Лабораторной работы №6 был успешно проведён полный цикл улучшения UX проекта. Приложение оценено по шести атрибутам стандарта ISO/IEC 25010:2011, что позволило выявить критические недочёты в области защиты от ошибок, обучаемости и доступности.

В результате внедрения клиентской валидации форм, toast-уведомлений, состояний Empty State и полной поддержки клавиатурной навигации по стандарту WCAG 2.0 юзабилити системы значительно улучшилось. Приложение стало более безопасным в использовании, интуитивно понятным для новых пользователей и инклюзивным. Трассируемость всех изменений задокументирована через GitHub Projects, что подтверждает соответствие команды процессам профессиональной разработки ПО.
